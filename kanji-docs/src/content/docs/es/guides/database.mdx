## Descripción General

Kanji soporta dos adaptadores de base de datos vía `@kanjijs/store`:

- **PostgreSQL** vía Drizzle ORM — Recomendado para la mayoría de proyectos
- **MongoDB** vía adaptador nativo — Para datos orientados a documentos

Ambos comparten una interfaz común `Database` y `QueryBuilder`. Los services escritos con la API común funcionan en **cualquiera de las dos bases** sin cambios.

## Configuración de PostgreSQL

### 1. Instalar Dependencias

```bash
bun add drizzle-orm postgres
bun add -d drizzle-kit @types/pg
```

### 2. Configurar StoreModule

```typescript
// src/app.module.ts
import { KanjijsModule } from '@kanjijs/core';
import { StoreModule } from '@kanjijs/store';
import { env } from '@kanjijs/common';
import { z } from 'zod';
import { users, posts } from './database/schema';

@KanjijsModule({
  imports: [
    StoreModule.forRoot({
      type: 'postgres',
      connectionString: env('DATABASE_URL', z.string().url()),
      schema: { users, posts },  // Objetos de tabla Drizzle — requerido para Postgres
    }),
  ],
})
export class AppModule {}
```

:::caution[El schema es requerido para Postgres]
El objeto `schema` mapea nombres de tablas a definiciones de tablas Drizzle. Sin él, `db.query.users` no sabe qué tabla consultar. MongoDB no necesita esto — los nombres de colección se infieren de la query.
:::

### 3. Definí tu Schema

```typescript
// src/database/schema.ts
import { pgTable, uuid, varchar, text, timestamp, boolean } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  name: varchar('name', { length: 256 }).notNull(),
  email: varchar('email', { length: 256 }).notNull().unique(),
  active: boolean('active').notNull().default(true),
  createdAt: timestamp('created_at').notNull().defaultNow(),
  updatedAt: timestamp('updated_at').notNull().defaultNow(),
});

export const posts = pgTable('posts', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id')
    .notNull()
    .references(() => users.id, { onDelete: 'cascade' }),
  title: varchar('title', { length: 512 }).notNull(),
  content: text('content'),
  published: boolean('published').notNull().default(false),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

// Inferí tipos desde el schema de Drizzle — sin interfaces manuales
export type User = typeof users.$inferSelect;
export type InsertUser = typeof users.$inferInsert;
```

### 4. Generar Migraciones

```bash
kanji migrate:generate    # Crea archivo de migración a partir de cambios en el schema
kanji migrate             # Aplica migraciones pendientes
```

## Configuración de MongoDB

### 1. Configurar StoreModule

```typescript
StoreModule.forRoot({
  type: 'mongodb',
  connectionString: env('MONGODB_URI', z.string().url()),
  dbName: env('MONGODB_DB', z.string()),
})
```

No se necesita el parámetro `schema`. Los nombres de colección se infieren de `db.query.collectionName`.

### 2. Definí Interfaces TypeScript

MongoDB no usa schemas de Drizzle. Definí las formas de tus documentos en TypeScript:

```typescript
// src/database/types.ts
export interface UserDocument {
  id: string;         // Se mapea a _id automáticamente
  name: string;
  email: string;
  active: boolean;
  createdAt: Date;
  updatedAt: Date;
}

export interface PostDocument {
  id: string;
  userId: string;
  title: string;
  content?: string;
  published: boolean;
  createdAt: Date;
}
```

### Comportamiento Específico de MongoDB

El adaptador maneja esto automáticamente:

- **Mapeo `id` ↔ `_id`** — Siempre usá `id` en tu código. El adaptador convierte de/ hacia `_id` de MongoDB de forma transparente.
- **Conversión de ObjectId** — Strings hexadecimales de 24 caracteres pasados en `where()` se auto-convertir a `ObjectId`.
- **Conexión lazy** — MongoDB se conecta en la primera query, no al iniciar.
- **Sin `RETURNING`** — Los insert/update/delete de MongoDB no retornan filas afectadas de la misma manera. El adaptador re-obtiene documentos después de la mutación para simular el comportamiento de `returning()`.

## Queries — API Común

Ambos adaptadores implementan la misma interfaz `QueryBuilder`. Este código funciona en **ambas** bases de datos, PostgreSQL y MongoDB:

```typescript
import { Injectable, Inject } from '@kanjijs/core';
import { DATABASE_CLIENT, Database } from '@kanjijs/store';

@Injectable()
export class UsersService {
  constructor(@Inject(DATABASE_CLIENT) private db: Database) {}

  // SELECT — buscar todos
  async findAll() {
    return this.db.query.users.select();
  }

  // SELECT — buscar por ID
  async findById(id: string) {
    return this.db.query.users.findById(id);
  }

  // SELECT — buscar por campo
  async findByEmail(email: string) {
    return this.db.query.users.findBy({ email });
  }

  // SELECT — con filtros, paginación, ordenamiento
  async findActive(page: number, limit: number) {
    const offset = (page - 1) * limit;
    return this.db.query.users
      .select()
      .where({ active: true })
      .limit(limit)
      .offset(offset)
      .orderBy('createdAt', 'desc');
  }

  // SELECT — campos específicos
  async findNames() {
    return this.db.query.users.select(['name', 'email']);
  }

  // INSERT — uno solo
  async create(data: { name: string; email: string }) {
    const [user] = await this.db.query.users.insert(data);
    return user;
  }

  // INSERT — varios
  async createMany(users: Array<{ name: string; email: string }>) {
    return this.db.query.users.insert(users);
  }

  // UPDATE
  async update(id: string, data: { name?: string }) {
    const [updated] = await this.db.query.users
      .where({ id })
      .update(data);
    return updated || null;
  }

  // DELETE
  async delete(id: string) {
    const results = await this.db.query.users.where({ id }).delete();
    return results.length > 0;
  }

  // TRANSACTION
  async transferData(fromId: string, toId: string) {
    return this.db.transaction(async (trx) => {
      const [from] = await trx.query.users.where({ id: fromId }).update({ active: false });
      const [to] = await trx.query.users.where({ id: toId }).update({ active: true });
      return { from, to };
    });
  }
}
```

### Referencia de la API QueryBuilder

| Método | Descripción | Ejemplo |
|---|---|---|
| `.select()` | Obtener todos los registros | `db.query.users.select()` |
| `.select(['campo'])` | Obtener campos específicos | `db.query.users.select(['name'])` |
| `.findById(id)` | Buscar por clave primaria | `db.query.users.findById('abc')` |
| `.findBy({ campo: valor })` | Buscar por campo | `db.query.users.findBy({ email: 'a@b.com' })` |
| `.where({ campo: valor })` | Filtrar registros | `db.query.users.where({ active: true })` |
| `.limit(n)` | Limitar resultados | `.limit(10)` |
| `.offset(n)` | Saltar resultados | `.offset(20)` |
| `.orderBy(campo, dir)` | Ordenar resultados | `.orderBy('createdAt', 'desc')` |
| `.insert(data)` | Insertar registro(s) | `db.query.users.insert({ name: 'Alice' })` |
| `.where({}).update(data)` | Actualizar registros | `.where({ id }).update({ name: 'Bob' })` |
| `.where({}).delete()` | Borrar registros | `.where({ id }).delete()` |
| `.transaction(fn)` | Ejecutar en transacción | `db.transaction(async (trx) => { ... })` |

:::note[El cláusula where es intencionalmente simple]
El método `where()` acepta objetos simples: `{ campo: valor }`. Esto funciona de manera idéntica en Postgres y MongoDB. Para queries complejas (OR, LIKE, condiciones anidadas), usá el método `raw()` con sintaxis específica de la base de datos.
:::

## Repository Pattern

Kanji usa el **Repository Pattern** para separar la lógica de acceso a datos de la lógica de negocio. El CLI genera repositories automáticamente cuando scaffoldeás un resource con `kanji g resource`.

### Flujo de Arquitectura

```
Controller → Service → Repository → Database
  (HTTP)     (negocio)  (acceso a datos)  (Postgres/MongoDB)
```

Cada capa tiene una única responsabilidad:

| Capa | Decorador | Responsabilidad |
|---|---|---|
| **Controller** | `@Controller()` | Ruteo HTTP, request/response, validación |
| **Service** | `@Injectable()` | Lógica de negocio, orquestación |
| **Repository** | `@Repository()` | Acceso a datos, queries, mutaciones |

### Crear un Repository

El decorador `@Repository()` marca una clase como proveedora de acceso a datos. Es semánticamente idéntico a `@Injectable()` pero comunica la responsabilidad de base de datos.

```typescript
// src/products/product.repository.ts
import { Repository, Inject } from '@kanjijs/core';
import { DATABASE_CLIENT, type Database } from '@kanjijs/store';
import type { ProductResponse, CreateProductInput } from './product.contracts.js';

@Repository()
export class ProductRepository {
  constructor(
    @Inject(DATABASE_CLIENT)
    private readonly db: Database,
  ) {}

  async create(input: CreateProductInput): Promise<ProductResponse> {
    const id = crypto.randomUUID();
    await this.db.query.products.insert({ id, ...input });
    return { id, ...input };
  }

  async findAll(): Promise<ProductResponse[]> {
    return this.db.query.products.select() as Promise<ProductResponse[]>;
  }

  async findOne(id: string): Promise<ProductResponse | null> {
    const result = await this.db.query.products.findById(id);
    return result as ProductResponse | null;
  }

  async update(id: string, input: Partial<CreateProductInput>): Promise<ProductResponse> {
    await this.db.query.products.where({ id }).update(input);
    return { id, ...input } as ProductResponse;
  }

  async delete(id: string): Promise<{ success: boolean }> {
    await this.db.query.products.where({ id }).delete();
    return { success: true };
  }
}
```

### Registrar en el Módulo

Los repositories van en el array `repositories` del módulo — separado de `providers` por claridad semántica:

```typescript
// src/products/product.module.ts
import { KanjijsModule } from '@kanjijs/core';
import { ProductController } from './product.controller.js';
import { ProductService } from './product.service.js';
import { ProductRepository } from './product.repository.js';

@KanjijsModule({
  controllers: [ProductController],
  repositories: [ProductRepository],   // acceso a datos
  providers: [ProductService],         // lógica de negocio
  exports: [ProductService],
})
export class ProductModule {}
```

### El Service inyecta el Repository

Los services nunca tocan la base de datos directamente — delegan al repository:

```typescript
// src/products/product.service.ts
import { Injectable } from '@kanjijs/core';
import { ProductRepository } from './product.repository.js';
import type { ProductResponse, CreateProductInput } from './product.contracts.js';

@Injectable()
export class ProductService {
  constructor(private readonly repository: ProductRepository) {}

  async create(input: CreateProductInput): Promise<ProductResponse> {
    // La lógica de negocio va acá (validación, efectos secundarios, etc.)
    return this.repository.create(input);
  }

  async findAll(): Promise<ProductResponse[]> {
    return this.repository.findAll();
  }

  async findOne(id: string): Promise<ProductResponse | null> {
    return this.repository.findOne(id);
  }
}
```

### Generar con el CLI

El comando `kanji g resource` genera el stack completo Controller → Service → Repository:

```bash
kanji g resource products --db=postgres --crud=create,findAll,findOne,update,delete
```

Esto crea:

```
src/products/
├── product.module.ts        # Módulo con los tres registrados
├── product.controller.ts    # Handlers HTTP
├── product.service.ts       # Lógica de negocio (inyecta Repository)
├── product.repository.ts    # Acceso a datos (inyecta Database)
├── product.contracts.ts     # Schemas Zod para validación
├── product.policy.ts        # Reglas de autorización
└── index.ts                 # Barrel exports
```

### Repository vs QueryBuilder directo en Service

| Enfoque | Cuándo usarlo |
|---|---|
| **Repository pattern** | La mayoría de proyectos — separación limpia, testeable, intercambiable |
| **`@Inject(DATABASE_CLIENT)` directo en Service** | CRUD simple sin lógica de negocio, o cuando el service ES la capa de datos |

:::tip[Regla general]
Si tu service tiene más de 2-3 métodos que solo reenvían a `this.db.query.*`, extraé un repository. El service debe contener lógica de negocio, no plumbing de SQL/NoSQL.
:::

### Repository con MongoDB

El mismo patrón funciona con MongoDB — solo cambia la sintaxis de query:

```typescript
@Repository()
export class ProductRepository {
  constructor(
    @Inject(DATABASE_CLIENT)
    private readonly db: Database,
  ) {}

  async create(input: CreateProductInput): Promise<ProductResponse> {
    const id = crypto.randomUUID();
    await this.db.collection('products').insertOne({ _id: id, ...input });
    return { id, ...input };
  }

  async findAll(): Promise<ProductResponse[]> {
    const docs = await this.db.collection('products').find().toArray();
    return docs.map(doc => ({ id: doc._id.toString(), ...doc })) as ProductResponse[];
  }
}
```

## Queries Exclusivas de PostgreSQL (Drizzle)

Cuando necesitás funcionalidades específicas de Drizzle (joins complejos, agregaciones, subqueries), podés usar Drizzle directamente:

```typescript
import { Injectable, Inject } from '@kanjijs/core';
import { DATABASE_CLIENT } from '@kanjijs/store';
import { eq, and, count, desc, like } from 'drizzle-orm';
import { users, posts } from '../../database/schema';

@Injectable()
export class UsersServicePostgres {
  constructor(@Inject(DATABASE_CLIENT) private db: PostgresJsDatabase<Record<string, never>>) {}

  // Join complejo
  async findWithPosts(userId: string) {
    return this.db.select()
      .from(users)
      .leftJoin(posts, eq(posts.userId, users.id))
      .where(eq(users.id, userId));
  }

  // Agregación
  async countByRole() {
    return this.db.select({
      role: users.role,
      count: count(),
    }).from(users).groupBy(users.role);
  }

  // Filtro complejo
  async search(term: string) {
    return this.db.select().from(users)
      .where(and(
        eq(users.active, true),
        like(users.name, `%${term}%`),
      ))
      .orderBy(desc(users.createdAt));
  }
}
```

:::warning[Portabilidad de base de datos]
Las queries específicas de Drizzle NO van a funcionar con MongoDB. Usá la API común de QueryBuilder para services que necesiten soportar ambas bases de datos.
:::

## Evitar Queries N+1

El problema N+1 ocurre cuando cargás entidades relacionadas en un loop:

```typescript
// ❌ MAL: Queries N+1 — 1 query para users + N queries para posts
const users = await this.db.query.users.select();
for (const user of users) {
  user.posts = await this.db.query.posts.where({ userId: user.id }).select();
}

// ✅ BIEN (Postgres): Query sola con join
const results = await this.db.select()
  .from(users)
  .leftJoin(posts, eq(posts.userId, users.id));

// ✅ BIEN (Independiente de la DB): Fetch en lote
const userIds = users.map(u => u.id);
const allPosts = await this.db.query.posts
  .select()
  .where({ userId: userIds[0] });  // Usá raw() para queries con $in

// ✅ BIEN (Paralelo): Fetch por usuario concurrentemente
const postsByUser = await Promise.all(
  users.map(u => this.db.query.posts.where({ userId: u.id }).select())
);
```

## Cambiar entre Bases de Datos

El único cambio es en `app.module.ts` — intercambiá la config de `StoreModule.forRoot()`:

```typescript
// PostgreSQL
StoreModule.forRoot({
  type: 'postgres',
  connectionString: env('DATABASE_URL', z.string().url()),
  schema: { users, posts },
})

// MongoDB (el mismo código de service funciona!)
StoreModule.forRoot({
  type: 'mongodb',
  connectionString: env('MONGODB_URI', z.string().url()),
  dbName: env('MONGODB_DB', z.string()),
})
```

El código de service que usa la API común de QueryBuilder requiere **cero cambios**.

## Migraciones

```bash
kanji migrate               # Ejecutar migraciones pendientes
kanji migrate:status        # Verificar estado de las migraciones
kanji migrate:rollback      # Revertir la última migración
kanji migrate:generate      # Generar nueva migración a partir de cambios en el schema
kanji db:push               # Push directo del schema (solo para desarrollo)
kanji db:studio             # Abrir Drizzle Studio (navegador visual de la DB)
```

:::caution[Orden de migraciones]
Nunca edites una migración ya commiteada. Siempre creá una nueva. Las migraciones fuera de orden causan errores de "relation already exists".
:::

## Errores Comunes

| Error | Síntoma | Solución |
|---|---|---|
| Falta `schema` en la config de Postgres | `Table 'x' not found in Drizzle schema` | Pasá el objeto `schema` a `StoreModule.forRoot()` |
| Usar `id` vs `_id` en MongoDB | La query retorna datos con `_id` en vez de `id` | Usá `id` en tu código — el adaptador mapea automáticamente |
| Imports de Drizzle con MongoDB | `eq is not a function` | Usá la API común de QueryBuilder, no imports de Drizzle |
| Queries N+1 | Endpoint lento a pesar de pocos datos | Usá joins (Postgres) o fetch en lote (ambas) |
| Editar migraciones commiteadas | "relation already exists" | Creá una nueva migración |
| Sin transacción en operaciones multi-paso | Datos parciales en caso de error | Usá `db.transaction()` |
