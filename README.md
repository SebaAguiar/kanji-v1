# Kanji Framework

Kanji is a modern, high-performance backend framework built for **Bun** using **Hono**, designed for modularity, type-safety, and secure-by-default architectures.

## Key Features

- **Modular Dependency Injection**: Deterministic container system inspired by modern frameworks, built on top of TypeScript decorators.
- **Contract-First Validation**: Declare API contracts using Zod schema models once, and gain automatic request validation, type checking, and OpenAPI documentation.
- **Database Agnostic Store**: Integrated support for PostgreSQL (via Drizzle ORM) and native MongoDB adapters.
- **Action-Level Security**: Native support for Class-Level Permissions (CLP) and Access Control Lists (ACL) mapped directly to Hono request contexts.
- **Developer Scaffolding CLI**: Interactive CLI wizard to generate modules, CRUD endpoints, policies, and test suites with zero manual setup.

## Quick Start

### Installation

Clone the repository and install dependencies using `pnpm`:

```bash
pnpm install
```

### Running Tests

Execute the global unit and integration test suite using Bun's native test runner:

```bash
bun test
```

## CLI Interactive Wizard

Kanji provides an interactive scaffolding wizard to boost productivity and enforce architectural conventions.

### Generate Resource Modules

To scaffold a complete, type-safe CRUD module (Contracts, Repository, Service, Controller, Module, and Tests):

```bash
cd examples/basic
bun ../../packages/cli/src/cli.ts g
```

*The generator automatically scans your `app.module.ts` to detect your active database adapter (PostgreSQL or MongoDB) and skips redundant setup questions.*

### Configure Route Security

To protect your generated routes with granular permissions:

```bash
bun ../../packages/cli/src/cli.ts g-auth
```

1. Select the module you want to protect.
2. Select which specific CRUD actions (`Create`, `Read`, `Update`, `Delete`) to secure.
3. Choose the authorization model per action:
   - **Role-based (RBAC)**: Specify allowed roles (e.g. `admin`, `moderator`).
   - **Owner-based (ACL)**: Restrict access to the resource owner (`resource.userId === user.userId`).
4. The CLI creates a `{resource}.policy.ts` file and auto-registers it in your module providers and exports.

## Granular Policy Example

Here is an autogenerated policy securing different CRUD actions on a `products` resource individually:

```typescript
import { Injectable } from '@kanjijs/core';
import type { ResourcePolicy } from '@kanjijs/auth';
import type { Context } from 'hono';

@Injectable()
export class ProductPolicy implements ResourcePolicy {
  canRead(c: Context, resource: any, user: any): boolean {
    return true; // Read is public
  }

  canCreate(c: Context, resource: any, user: any): boolean {
    const allowed = ["admin"];
    return user.roles.some((role: string) => allowed.includes(role)); // RBAC
  }

  canUpdate(c: Context, resource: any, user: any): boolean {
    return resource.userId === user.userId || user.roles.includes('admin'); // ACL
  }

  canDelete(c: Context, resource: any, user: any): boolean {
    return resource.userId === user.userId || user.roles.includes('admin'); // ACL
  }
}
```

This policy is mapped to your controller using `@UseGuards(acl(...))` and `@UseGuards(clp(...))` decorators, validating security conditions at Hono runtime prior to handler execution.
